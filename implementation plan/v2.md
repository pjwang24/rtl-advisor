# RTL Advisor V2 Implementation Plan

## 1. Outcome and promotion criteria

V2 will turn the v1 experiment into a safety-first, pre-synthesis RTL advisor:

- Accept generated SystemVerilog, individual RTL files, or filelists.
- Detect and rank transformations from the existing nine optimization families.
- Use a calibrated, interpretable gate to make the final `recommend` or `abstain` decision.
- Use GPT-5.6 Sol at xhigh effort to explain the decision, source location, expected PPA direction, and verification requirements.
- Optionally generate isolated RTL candidates; never modify the original source.
- Require lint and Yosys formal equivalence before labeling any generated candidate valid.
- Train and benchmark a random-forest challenger, but defer any live ML promotion to v2.1.
- Use only generated RTL. Multiple SoC blocks and company RTL are not required.
- Target block/IP-sized synthesizable RTL. Full CSS/SoC signoff, proprietary frontend support, and new optimization families are outside v2.

V2 is promoted only if the untouched 72-case blind suite meets all primary gates:

- Macro and micro actionable accuracy: at least 65%.
- Improvement over v1 hybrid xhigh: at least 10 percentage points, with the lower bound of a paired 95% bootstrap confidence interval above zero.
- Harmful recommendation rate: at most 10% of recommendations.
- Opportunity coverage: recommendations on at least 50% of cases that contain an eligible optimization.
- PPA direction accuracy: at least 70% at 90% or greater direction coverage.
- Exact-best candidate accuracy: at least 60% of recommended positive cases.
- Normalized ranking regret: at most 0.25 and at least 30% below v1 hybrid.
- Every emitted accepted candidate passes lint and formal equivalence.

If the blind suite misses a target, report the result without retuning. Corrections move to v2.1 with a newly generated blind suite.

## 2. Architecture and public interfaces

### Analysis flow

```text
RTL/files/filelist
        │
        ├── PySlang parse, elaborate, lint, and source mapping
        ├── Yosys pre-mapping structural graph
        │
        ▼
Registered pattern rules → candidate descriptors → calibrated safety gate
                                                   │
                         ┌─────────────────────────┴─────────────────────────┐
                         ▼                                                   ▼
              deterministic result                               optional Codex explanation
                         │                                                   │
                         └────────────────── fixed decision ─────────────────┘
                                                   │
                              optional isolated candidate generation
                                                   │
                                       lint → formal equivalence
```

PySlang is appropriate because it supports SystemVerilog parsing, elaboration, source-preserving syntax trees, Python bindings, and refactoring. Pin `pyslang==9.1.0`, which has a CPython 3.13 macOS ARM64 wheel and includes Python `SyntaxRewriter` bindings. [PySlang package](https://pypi.org/project/pyslang/9.1.0/), [Slang changelog](https://github.com/MikePopoloski/slang/blob/master/CHANGELOG.md).

### CLI

Preserve the existing manifest command unchanged and add:

```text
rtl-advisor analyze-rtl
  --top <module>
  (--file <path>... | --filelist <path>)
  [-I <include-dir>...]
  [-D <name[=value]>...]
  [--profile balanced|timing-first|area-first]
  [--mode calibrated|advisor]
  [--output <directory>]
  [--emit-candidates]
  [--candidate-source templates|templates+codex]
  [--allow-model-source]
```

Defaults:

- `--profile balanced`
- `--mode calibrated`
- `--candidate-source templates`
- Read-only analysis unless `--emit-candidates` is supplied.
- Codex source excerpts may leave the machine only when `--allow-model-source` is explicitly supplied.
- Codex-generated RTL requires all three: `--emit-candidates`, `--candidate-source templates+codex`, and `--allow-model-source`.

Filelists support source paths, nested `-f`, `-I`, `+incdir+`, `-D`, and `+define+`. Unsupported tool-specific arguments produce a precise input error.

Exit codes:

- `0`: completed analysis, including `recommend`, `abstain`, or `unsupported`.
- `2`: invalid input, parsing, or configuration.
- `3`: required tool execution failure.
- `4`: analysis completed, but explicitly requested candidate generation produced no verified candidate.

### Versioned data contracts

Add `DesignInputV2` containing:

- Ordered source files and filelist provenance.
- Top module, include directories, and defines.
- Resolved canonical paths and content hashes.
- Tool and configuration hashes.
- A design hash used as the artifact key.

Add `CandidateScoreV2` containing:

- Candidate and transformation IDs.
- Optimization family and source locations.
- Required rule preconditions.
- Predicted delay, area, and cell-count improvements.
- Calibrated prediction intervals.
- Profile eligibility, conservative utility, rank, and rejection reasons.
- Candidate-generation and verification status.

Add `AnalysisResultV2` containing:

- Schema version and design hash.
- `decision: recommend | abstain | unsupported`.
- Selected profile and candidate ID.
- All scored candidates.
- Gate model version/hash and decision reasons.
- Codex explanation status and text.
- Lint, formal, and tool diagnostics.
- Artifact paths.

Artifacts are written under a design-hash directory. Original RTL is never changed.

### Safety gate

Define positive improvements as:

```text
delay_delta = 100 × (baseline_delay - candidate_delay) / baseline_delay
area_delta  = 100 × (baseline_area  - candidate_area)  / baseline_area
cell_delta  = 100 × (baseline_cells - candidate_cells) / baseline_cells
```

The live gate uses the conservative lower bound for every delta:

| Profile | Candidate eligibility | Ranking utility |
|---|---|---|
| Balanced | `(delay ≥ 3% and area ≥ -10%)` or `(area ≥ 5% and delay ≥ -2%)` | `delay + area + 0.1 × cells` |
| Timing-first | `delay ≥ 3% and area ≥ -20%` | `2 × delay + 0.5 × area + 0.05 × cells` |
| Area-first | `area ≥ 5% and delay ≥ -10%` | `0.5 × delay + 2 × area + 0.1 × cells` |

The highest conservative utility wins. Ties are broken by cell improvement and then stable candidate ID. The gate abstains when:

- No candidate conservatively clears the profile.
- The family or transformation is unsupported.
- A feature is outside the calibration envelope.
- Source mapping is ambiguous.
- Required model or feature-schema hashes do not match.
- Parsing, structural extraction, lint, or formal verification is inconclusive.

Codex cannot override the decision or selected candidate. A schema-inconsistent explanation is discarded while the deterministic gate result remains valid.

## 3. Implementation sequence

### Phase 0 — Freeze the v2 contract

- Save this plan as `implementation plan/v2.md`.
- Add versioned configuration for feature schema, profiles, models, prompts, suites, and tool versions.
- Add optional dependencies:
  - `sv`: `pyslang==9.1.0`
  - `train`: `scikit-learn==1.9.0`
- Keep existing v1 functionality usable without either extra.
- Record current v1 results and all 66 existing tests as the regression baseline.

### Phase 1 — Real RTL frontend

- Implement files/filelist normalization into `DesignInputV2`.
- Use PySlang for parsing, elaboration, diagnostics, syntax/source locations, and lint.
- Produce a Yosys graph before ABC or technology mapping for structural features.
- Correlate Yosys `src` attributes with PySlang syntax locations.
- Return `unsupported` if PySlang accepts constructs that the Yosys pre-mapping frontend cannot process.
- Ensure analysis never reads post-synthesis metrics or blind labels.

### Phase 2 — Topology-diverse generated corpus

Generate 48 cases for each existing family: 40 calibration and 8 blind, totaling 360 calibration and 72 blind cases.

Each case contains:

- `v0`: baseline.
- `v1`, `v2`, `v3`: logically equivalent registered alternatives.
- `n0`: deliberately inequivalent negative control.

Expand topology across these dimensions:

| Family | Topology dimensions |
|---|---|
| Resource sharing | add/sub/multiply, width, signedness, 2–4 branches, common operand side |
| Adder association | 4/6/8/12 operands, width, signedness, flat/skewed input depth |
| Priority logic | 4/8/12/16 requests, data width, priority direction, default behavior |
| Mux placement | operation, width, 2–4 inputs, common operand side, signedness |
| Decode factoring | opcode width, match count, repeated-use count, exact/masked comparisons |
| Comparator structure | relation, width, signedness, fanout, constant shape |
| Variable shift | direction, data width, shift-width excess, guard style, arithmetic/logical |
| Width/signedness | operation, source width, extension ratio, signedness mix, truncation |
| Popcount | width, exact/threshold/saturating use, linear/chunked/tree structure |

Selection is deterministic:

1. Enumerate legal parameter tuples.
2. Greedily choose the tuple covering the most previously uncovered parameter-value pairs.
3. Break ties using SHA-256 of the family, tuple, and seed `20260714`.
4. Select 48 tuples per family.
5. Choose eight blind tuples using the same coverage-maximizing process; the other 40 become calibration.
6. Freeze all descriptors, RTL hashes, and split membership before synthesis.

Reject corpus generation unless:

- Every legal parameter value appears in calibration.
- Every parameter value represented by the selected 48 appears in blind where eight cases permit it.
- At least 85% of feasible pairwise value interactions are covered.
- PySlang and Verilator lint pass for `v0`–`v3`.
- Yosys equivalence passes for `v1`–`v3`.
- Yosys equivalence rejects every `n0`.

V1 cases remain a legacy regression suite and are not used for v2 training.

### Phase 3 — Ground truth and feature extraction

For `v0`–`v3`, run the existing pinned Yosys/ABC/Nangate45 flow at the fixed 10 ns constraint. Record delay, area, cell count, tool logs, and hashes.

Extract only pre-mapping features:

- Operator counts and widths.
- Arithmetic, comparator, mux, and control depth.
- Fan-in/fanout distributions.
- Branch and request counts.
- Dynamic shift structure.
- Signedness and extension conversions.
- Repeated expression fingerprints.
- Local cone size/depth and source span.
- Candidate family and transformation ID.

Training, feature selection, and threshold calibration may access only the 360 calibration cases.

### Phase 4 — Interpretable gate and ML challenger

The calibrated gate consists of registered rules plus 27 shallow regression trees: one per family and PPA metric.

Gate tree configuration:

- `DecisionTreeRegressor`
- Maximum depth `3`
- Minimum leaf size `8`
- Random seed `20260714`
- Candidate transformation ID encoded in the fixed feature vector
- Five-fold cross-validation grouped by case/topology signature

Use out-of-fold residuals to construct 90% conformal intervals for each family, transformation, and metric. Retrain on all calibration cases and export trees, intervals, feature ordering, and calibration envelopes to deterministic JSON. Live evaluation reads this JSON and does not require scikit-learn.

Train the challenger separately:

- `RandomForestRegressor`
- 500 trees
- Maximum depth `8`
- Minimum leaf size `3`
- `max_features="sqrt"`
- Bootstrap enabled
- Random seed `20260714`
- Five grouped folds
- Multi-output delay, area, and cell targets

The challenger is benchmark-only. Store its trusted local artifact with a SHA-256 hash and reject unrecognized model files. Its v2 results become evidence for a v2.1 decision, never an automatic promotion.

### Phase 5 — Candidate generation

- Rules first produce up to three ranked, registered candidates.
- Apply one source transformation per candidate.
- Use PySlang source ranges and `SyntaxRewriter` for deterministic templates.
- Write candidates into isolated artifact workspaces with copied filelists.
- Never edit source files in place.
- If deterministic application fails, abstain unless explicit Codex fallback was enabled.
- Codex fallback receives only the selected local source span, surrounding declarations, compile context, and the fixed transformation contract.
- Reject candidates that introduce unrelated edits or touch more than the selected transformation span.

Verification order:

1. PySlang compile and lint.
2. Verilator lint when the unmodified baseline passes the same invocation.
3. Whole-design Yosys equivalence using the same top/files/defines.
4. `equiv_simple`, followed by induction for unresolved sequential points.
5. `equiv_status -assert`.

Any timeout, unsupported construct, or inconclusive proof rejects the candidate. Simulation alone can never approve a candidate.

### Phase 6 — Codex advisor

Use GPT-5.6 Sol with xhigh effort.

The prompt contains:

- Blinded synthetic RTL or explicitly authorized live excerpts.
- Pre-mapping structural summary.
- Registered candidate descriptions.
- Gate decision, prediction intervals, and abstention reasons.
- No measured synthesis result, blind label, or candidate identity that leaks ground truth.

The response schema permits explanation, risks, source references, predicted metric directions, and verification guidance. It does not permit changing the gate decision or candidate ID.

## 4. Evaluation protocol

### Six benchmark arms

Evaluate all 72 blind cases with:

1. V1 rules.
2. V1 Codex xhigh.
3. V1 hybrid xhigh.
4. V2 calibrated gate.
5. V2 safe advisor xhigh.
6. V2 random-forest challenger.

Balanced is the frozen headline profile. Timing-first and area-first are secondary analyses computed from the same measurements without additional model calls.

### Fixed 264-call model budget

- 72 calls: v1 Codex.
- 72 calls: v1 hybrid.
- 72 calls: v2 advisor.
- 48 stability calls:
  - Select one stable-hash case from every family.
  - Add a second case from resource sharing, mux placement, and priority logic.
  - Run two additional repetitions for both v1 hybrid and v2 advisor.

Total: `216 + 48 = 264` GPT-5.6 Sol xhigh calls.

Benchmark model failures are recorded without retries so the budget and reliability measurement remain fixed.

### Metrics

Report:

- Actionable accuracy:
  - Recommend an eligible candidate when one exists.
  - Abstain when none exists.
- Harmful recommendation rate:
  - Ineligible recommendations divided by all recommendations.
- Opportunity coverage:
  - Recommended eligible cases divided by all cases containing an eligible candidate.
- Exact-best candidate accuracy and normalized ranking regret.
- Delay, area, and cell direction accuracy using a ±1% neutral band.
- Direction coverage.
- Per-family macro metrics and whole-suite micro metrics.
- Candidate generation yield, lint yield, formal pass/rejection rates.
- Advisor schema validity, gate consistency, latency, token usage, and explanation stability.

Use 10,000 paired bootstrap resamples stratified by family with seed `20260714`. Publish 95% percentile confidence intervals.

### Blind governance

Before the first blind run, create a benchmark lock containing hashes for:

- Calibration and blind case descriptors.
- Feature schema and model artifacts.
- Profile thresholds and utilities.
- Prompt and response schemas.
- Tool versions.
- Benchmark arms, call allocation, and statistical procedure.

The blind runner refuses to start if any hash differs. It records the unseal timestamp and allows only the preregistered run. Subsequent executions are labeled reruns, not fresh blind evaluations.

### OpenROAD cross-check

After the primary blind evaluation, cross-check 27 cases:

- Two calibration cases per family: lowest and median stable hash.
- One blind case per family: lowest stable hash.
- Four variants per case: 108 physical-design runs.

Use the official OpenROAD-flow-scripts Docker flow, pinned to the immutable image digest captured at bootstrap, with Nangate45 and detailed routing. [ORFS Docker documentation](https://github.com/The-OpenROAD-Project/OpenROAD-flow-scripts/blob/master/docs/user/BuildWithDocker.md).

For each case:

- Use a 10 ns virtual clock with consistent input/output constraints.
- Compute one fixed square die for all four variants from the largest Yosys mapped area at 35% utilization.
- Round the die side to the next 10 µm with a 100 µm minimum.
- Use a 10 µm core margin.
- Record post-route timing, standard-cell area/count, wirelength, congestion, and DRCs.

Physical validation requires:

- At least 24 of 27 complete four-variant case sets.
- At least two complete case sets per family.
- At least 80% Yosys/OpenROAD action-label agreement.
- At least 75% per-metric direction agreement.

OpenROAD remains a secondary validation layer and cannot retroactively change the frozen Yosys blind labels.

## 5. Tests, artifacts, and intervention points

Required tests include:

- All existing 66 v1 tests.
- CLI/filelist/include/define normalization and hashing.
- PySlang diagnostics and source-location correlation.
- Generator determinism, marginal coverage, and pairwise coverage.
- Positive equivalence and negative-control rejection for every family.
- Exact profile-boundary and utility-ranking tests.
- Feature extraction tests proving no mapped metrics or blind labels are accessible.
- Grouped-fold leakage tests.
- Gate determinism, calibration interval, OOD, and abstention tests.
- Random-forest reproducibility.
- Source-preserving deterministic rewrite tests.
- Tests proving original RTL is byte-identical after candidate generation.
- Mocked Codex responses attempting to override the gate.
- Lint/formal timeout and inconclusive-proof rejection.
- Benchmark-lock tamper detection and exact 264-call accounting.
- OpenROAD metric parsing, incomplete-run handling, and fixed-floorplan checks.
- End-to-end generated case, live file, filelist, advisor, and candidate smoke tests.

Final artifacts:

- `implementation plan/v2.md`
- `progress updates/july 14th.md`
- Machine-readable calibration, benchmark, stability, and OpenROAD reports
- A concise comparison of v1 rules, v1 Codex, v1 hybrid, v2 gate, v2 advisor, and the ML challenger
- Explicit `promoted` or `not promoted` status with no post-blind retuning

Expected user intervention is limited to:

- Starting or approving Docker Desktop before the OpenROAD phase if the daemon remains unavailable.
- Supplying or confirming model credentials before the 264-call benchmark if credentials are unavailable.

All other implementation and local verification proceeds continuously.
